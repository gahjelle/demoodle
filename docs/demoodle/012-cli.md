# W12 — CLI Front End

## The Imperative Shell

The CLI is Demoodle's **imperative shell**: the thin outer layer that handles user
input, filesystem I/O, and side effects, while delegating all computation to the
functional core.

`frontends/cli.py` does four things:
1. Reads config from disk (via `configaroo`)
2. Constructs pipeline stages from config and architecture
3. Calls `runner.run()` to execute the graph
4. Prints results to stdout

No model math, no tokenization logic, no caching decisions live here. The shell is
intentionally thin so it can be replaced — the TUI (W29) and web front end (W30)
call the same `runner.run()` with the same stages and produce different output.

## `train` and `call`: the two modes

Every language model system exposes two fundamental operations:
- **Train**: consume data, adjust weights, produce a policy
- **Generate**: consume a policy and a prompt, produce text

HuggingFace CLI exposes these as separate `run` subcommands for fine-tuning and
inference. The OpenAI platform splits them across the fine-tuning API and the
completions/chat API. Demoodle's `demoo train` and `demoo call` are the same
split, at the smallest possible scale.

`call` invokes `train()` internally to ensure the pipeline is up to date before
generating. On a cache hit, `train()` completes in milliseconds. This mirrors how
inference servers start by loading a checkpoint — the checkpoint is the product of
training, and loading it is essentially free compared to generating.

## Caching as a user-facing feature

The training output distinguishes a fresh run from a cache hit:

```
Training  step 5000/5000 • loss: 2.3142   ← fresh run with live sparkline
Done      step 5000/5000 • loss: 2.3142

Cached    step 5000/5000 • loss: 2.3142   ← cache hit, instant
```

Most systems hide caching from the user entirely. Demoodle surfaces it because the
cache is one of the educational points — the same inputs always produce the same
outputs, and the system knows it. This is the same UX principle as `make` or
Bazel: show the user what was rebuilt and what was skipped, so the caching model
is visible rather than magic.

## The two-phase pipeline

Constructing the architecture requires knowing `vocab_size`, which is only known
after the tokenizer is trained. The CLI handles this with a two-phase run:

```python
phase1 = runner.run([tok_stage], {"corpus": corpus}, rng, cache_dir)
vocab_size = phase1["tokenizer"].vocab_size

arch = _make_arch(vocab_size)
return runner.run([tok_stage, dataset_stage, pretrain_stage], ...)
```

Phase 1 runs (or loads from cache) the tokenizer stage alone. Phase 2 runs the
full graph with the correctly-sized architecture. The tokenizer stage appears in
both calls but hits the cache immediately in phase 2.

This data-dependent initialization pattern appears in real systems when schema or
configuration must be inferred from data before the main computation can be
structured — a vocabulary that grows with the corpus, a feature dimension that
depends on the dataset, a model shape that adapts to input size.
