# T3 — Random Number Generation

**What T3 builds:** `trainadillo/_rng.py` — a `Generator` class and module-level
`manual_seed()`, matching PyTorch's interface while enforcing explicit seeding.

---

## Why ML needs reproducible randomness

Training a neural network involves randomness at several points: weight
initialization, data shuffling, dropout masks, and sampling. If you can't reproduce
a run, you can't reason about it:

- **Debugging**: "Loss spiked at step 3000" — did the model diverge, or was the
  data shuffle unlucky? With a fixed seed you can replay the exact run.
- **Ablations**: "Does adding dropout help?" — you need the same randomness in
  both runs to isolate the variable.
- **Paper reproduction**: A published result is only reproducible if the seed is
  reported. Many ML papers list seeds alongside hyperparameters for this reason.

The flip side: silent non-reproducibility is one of ML's most common footguns.
You run an experiment, get a good number, re-run it, get a different number —
and you don't know which to trust, or whether the first was just lucky.

## Two schools of randomness

### PyTorch: global mutable state

```python
torch.manual_seed(42)     # seeds a global CPU generator
x = torch.rand(5)         # draws from the global generator
y = torch.rand(5)         # advances the global generator further
```

Convenient. One call sets up reproducibility for everything below it.

The problem: any code that uses randomness advances the global sequence. Library
code, data loaders, preprocessing pipelines — if any of them call a random function,
they perturb your model's sequence. Adding a `print(torch.rand(1))` for debugging
changes every subsequent draw. Code is fragile to insertion order.

PyTorch's answer: pass an explicit `generator=` to functions that support it. But
the global default always exists, so nothing *forces* you to.

### JAX: explicit keys everywhere

```python
key = jax.random.PRNGKey(42)     # a value, not global state
key, subkey = jax.random.split(key)
x = jax.random.normal(subkey, (5,))
```

No global state at all. Every random call takes an explicit key. Splitting a key
produces two independent child keys. The sequence is entirely determined by the
key you pass — library code can't perturb it because it never sees your key.

Rigid, but impossible to accidentally contaminate. The Demoodle `RNG` value type
(`core/rng.py`) follows this philosophy: an immutable seed that `.split()`s into
children, passed explicitly through every stage.

### Trainadillo: PyTorch API, JAX discipline

Trainadillo's API is shaped like PyTorch (for compatibility), but Demoodle uses it
with JAX discipline: seeds are always controlled explicitly. The `Generator` class
provides the PyTorch-compatible interface; the module-level default being `None`
until seeded enforces the discipline.

## The algorithm: PCG64

### What's in a PRNG?

A pseudo-random number generator (PRNG) is a deterministic function that takes a
seed and produces a sequence of numbers that *look* random. "Looking random" means
passing statistical tests: no detectable patterns, uniform distribution, long period
before repeating.

### Three algorithms in this space

| Algorithm                      | Used by                                              | Period | Notes                                                                          |
| ------------------------------ | ---------------------------------------------------- | ------ | ------------------------------------------------------------------------------ |
| **MT19937** (Mersenne Twister) | NumPy (old `np.random.*`), default in many languages | 2¹⁹⁹³⁷ | Large state (2.5KB), statistically ok, predictable after observing 624 outputs |
| **PCG64**                      | NumPy ≥1.17 default, trainadillo                     | 2¹²⁸   | Small state (16 bytes), better statistics, fast                                |
| **Philox**                     | PyTorch (GPU)                                        | 2¹²⁸   | Counter-based, trivially parallelisable across GPU threads                     |

PCG64 (Permuted Congruential Generator, 64-bit) is the right choice here because:
- It has better statistical properties than MT19937 (passes more randomness tests)
- It's smaller and faster
- NumPy 1.17+ recommends it as the default
- Trainadillo has no GPU, so Philox's parallelism advantage is irrelevant

### Why seeds don't reproduce PyTorch sequences

If you seed trainadillo with `42` and PyTorch with `42`, you get different numbers.
Same seed, different algorithm → different sequence. This is expected and not a bug.
Reproducibility is *within* trainadillo — the same trainadillo seed always produces
the same trainadillo sequence. Don't mix seeds across frameworks.

```python
# trainadillo
trainadillo.manual_seed(42)
print(trainadillo.rand(3))   # [0.773, 0.438, 0.859] (PCG64)

# PyTorch
torch.manual_seed(42)
print(torch.rand(3))         # [0.887, 0.117, 0.225] (Philox/MT19937)
```

## The module-level footgun

PyTorch initialises its default generator with OS entropy at import time:

```python
import torch
torch.rand(5)   # works immediately, but non-reproducible — OS entropy seed
```

Convenient, but the result changes every run. If you forget `torch.manual_seed(42)`,
your training run is different every time and you don't know why.

Trainadillo makes this loud instead:

```python
import trainadillo as torch
torch.rand(5)              # RuntimeError: no default seed set. Call manual_seed().
torch.manual_seed(42)
torch.rand(5)              # works, reproducible
```

The `_default_generator` module variable is `None` at import time. T4's `rand()` and
`randint()` check for `None` and raise with a message pointing to `manual_seed()`.
This forces the footgun to be a compile-time (test-time) error rather than a
silent runtime surprise.

The error is testable: Demoodle's test suite can verify that no code path draws
from the default generator without seeding it first.

## `Generator(unseeded)` as a debugging signal

```python
g = Generator()
print(g)   # Generator(unseeded)

g.manual_seed(42)
print(g)   # Generator(seeded)
```

The repr doesn't show the actual seed value (it's not stored after seeding — only
the numpy PRNG state is kept). It shows whether `manual_seed()` has been called.

A `Generator(unseeded)` in a print or debug session tells you: *this generator is
using OS entropy — results will differ every run*. A `Generator(seeded)` tells you:
*this generator was explicitly controlled — results are reproducible*.

This is deliberately different from a hypothetical `Generator(seed=42)` repr, which
would misleadingly suggest that `seed` is a constructor parameter. The words
`seeded`/`unseeded` are state labels, not parameter names.

## How Demoodle uses this

Demoodle's `RNG` value type (`core/rng.py`) is the source of truth for seeds in the
pipeline. Its `.generator()` method converts an `RNG` into a `trainadillo.Generator`:

```python
rng = RNG(seed=42)
g = rng.generator()   # trainadillo.Generator, seeded from rng.seed
```

This means:
- All random ops in Demoodle use explicit generators, never the module-level default
- The module-level `_default_generator` is never touched in production code
- Seeds flow through the pipeline deterministically via `RNG.split()`

The `RuntimeError` on unseeded `rand()` exists precisely to catch code that
accidentally bypasses this discipline — for example, a new architecture that calls
`trainadillo.rand()` directly instead of accepting an `RNG` and converting it.
